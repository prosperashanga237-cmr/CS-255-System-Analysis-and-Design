Project Overview

The DriverPass project focused on designing a system for DriverPass, a company that wanted to help students better prepare for DMV driving tests. The client needed an online system that would allow students to register for training packages, take practice tests, schedule driving lessons, track their progress, and access updated DMV information. The system also needed to support instructors, secretaries, administrators, and managers while maintaining security and ease of use.

Through this project, I created a business requirements document and a system design document that translated the client's needs into a complete system design. The project involved gathering requirements, analyzing user needs, creating UML diagrams, and designing technical solutions that support DriverPass's business goals.

What I Did Well

One area I performed particularly well was identifying and documenting the client's requirements. I carefully analyzed the DriverPass interview and business needs to ensure that all major user roles and system functions were represented in the design.

I also believe I did a strong job creating UML diagrams that illustrated how users interact with the system. The use case, activity, sequence, and class diagrams helped visualize the system and demonstrated how different components work together to meet the client's requirements.

What I Would Revise

If I could revise one part of my work, I would spend more time refining the UML diagrams to improve readability and organization. While the diagrams accurately represented the system, additional refinement could make them easier for developers and stakeholders to understand.

I would also add more exception paths and alternate scenarios to demonstrate how the system should respond when errors occur, such as failed payments, scheduling conflicts, or unavailable instructors.

Interpreting User Needs

Understanding user needs was one of the most important aspects of this project. I reviewed the client's requirements and identified the goals of each type of user, including students, instructors, administrators, secretaries, and managers.

I then translated those needs into system requirements and UML models that described how the software should function. Considering user needs is critical because a system can only be successful if it solves real problems for its users. A design that ignores user requirements may be technically correct but ultimately fail to provide value.

My Approach to Software Design

When designing software, I begin by understanding the client's goals and business requirements. I identify stakeholders, gather requirements, define system functionality, and create models that describe how the system should operate.

In future projects, I plan to continue using techniques such as requirement analysis, UML modeling, process mapping, prototyping, and stakeholder feedback. These strategies help ensure that system designs remain organized, scalable, and aligned with user expectations. I have learned that successful software design requires both technical planning and continuous communication with stakeholders throughout the development process.
